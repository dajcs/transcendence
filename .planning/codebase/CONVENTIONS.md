# Coding Conventions

**Analysis Date:** 2026-03-24

## Naming Patterns

**Files:**
- Frontend components: PascalCase, e.g., `UserProfile.tsx`, `BetCard.tsx`
- Frontend utilities/hooks: camelCase, e.g., `useAuth.ts`, `formatCurrency.ts`
- Backend routes: snake_case, e.g., `user_routes.py`, `bet_models.py`
- Backend utilities: snake_case, e.g., `auth_utils.py`, `validation.py`
- Database migration files: Generated by Alembic, e.g., `001_initial_schema.py`
- Test files: `[name].test.ts` (frontend), `test_[name].py` (backend)

**Functions:**
- Frontend: camelCase for all functions, hooks, and callbacks
  - Example: `handleBetSubmit()`, `calculateOdds()`, `useMarketData()`
- Backend: snake_case for all functions, route handlers, and utilities
  - Example: `validate_bet_input()`, `calculate_karma()`, `get_user_markets()`

**Variables:**
- Const declarations: camelCase for most values
- React state: camelCase for state names, `setState` format
  - Example: `const [isLoading, setIsLoading] = useState(false)`
- Backend constants: UPPER_SNAKE_CASE for module-level constants
  - Example: `MAX_BET_SIZE = 1000`, `JWT_EXPIRY_HOURS = 24`

**Types/Interfaces:**
- TypeScript: PascalCase for all interfaces, types, and enums
  - Example: `interface Market {}`, `type BetPosition = 'yes' | 'no'`, `enum UserRole {}`
- Python: PascalCase for class names, snake_case for module-level types in docstrings
  - Example: `class Market(Base):`, `class UserBet(Base):`

## Code Style

**Formatting:**
- Frontend: Prettier with 2-space indentation
  - Config: `.prettierrc` (to be created during scaffolding)
  - Line length: 100 characters
  - Semicolons: required
  - Trailing commas: all
- Backend: Black with 88-character line length (Python standard)
  - Config: `pyproject.toml` (under `[tool.black]`)
  - No semicolons in Python

**Linting:**
- Frontend: ESLint with `@typescript-eslint` rules
  - Config: `eslint.config.js`
  - Strict rules: no unused variables, no implicit `any`, enforce type annotations on functions
- Backend: Ruff with strict mode
  - Config: `pyproject.toml` under `[tool.ruff]`
  - Enforce: import sorting (isort), type hints where practical, docstring formats

## Import Organization

**Order:**
1. Node.js / Python standard library imports
2. Third-party framework imports (React, FastAPI, etc.)
3. Third-party utility imports (lodash, dateutil, etc.)
4. Local absolute imports (using path aliases)
5. Local relative imports

**Path Aliases:**

Frontend (TypeScript):
```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@/components/*": ["src/components/*"],
      "@/hooks/*": ["src/hooks/*"],
      "@/lib/*": ["src/lib/*"],
      "@/types/*": ["src/types/*"]
    }
  }
}
```

Backend (Python):
```python
# No formal alias system; use relative imports for local modules
# Example: from app.models import User
# from app.utils import validate_bet
```

## Error Handling

**Patterns:**

Frontend:
- Use try-catch for async operations and API calls
- Return user-friendly error messages; log technical details
- Always catch Promise rejections in async functions
- Use Zod for runtime validation of API responses
- Example:
  ```typescript
  try {
    const result = await api.placeBet(betData);
    return result;
  } catch (error) {
    console.error('Bet placement failed:', error);
    toast.error(error.message || 'Failed to place bet');
    throw error;
  }
  ```

Backend:
- Use FastAPI HTTPException for all API errors
- Include status codes and detail messages
- Log all errors with context (user ID, request details)
- Never expose internal implementation details in error responses
- Example:
  ```python
  from fastapi import HTTPException, status

  if not user:
      raise HTTPException(
          status_code=status.HTTP_404_NOT_FOUND,
          detail="User not found"
      )
  ```

## Logging

**Framework:**
- Frontend: `console` (no dedicated logging framework; use structured logs for debugging)
- Backend: Python `logging` module with configured handlers

**Patterns:**

Frontend:
- Use `console.log()` for development, remove before production
- Use `console.error()` for caught errors
- Prefix logs with context: `console.log('[BetCard]', 'Processing bet...')`

Backend:
- Use `logging.getLogger(__name__)` in each module
- Log levels: DEBUG (development), INFO (state changes), WARNING (recoverable issues), ERROR (failures)
- Always include relevant context: user ID, request ID, market ID
- Example:
  ```python
  logger = logging.getLogger(__name__)
  logger.info(f"Bet placed by user {user_id} on market {market_id}")
  ```

## Comments

**When to Comment:**
- Complex business logic (e.g., bet resolution calculations)
- Non-obvious algorithm or formula implementation
- Workarounds or temporary solutions
- Public API documentation (JSDoc/docstrings)
- Do NOT comment obvious code

## Function Design

**Size:**
- Keep functions under 30 lines; extract complexity into helpers
- Prefer composition over deeply nested logic

**Parameters:**
- Limit to 3-4 parameters; use objects/dataclasses for more
- Example (bad): `function bet(marketId, amount, position, timeout, strategy)`
- Example (good): `function bet(marketId: string, params: BetParams)`

**Return Values:**
- Always specify return types explicitly (TypeScript)
- Return data structures, not side effects
- Use async/await for async operations; avoid callback patterns

## Module Design

**Exports:**
- Named exports preferred; use default export only for page components
- Example: `export { calculateOdds, useMarketData }`
- Page components: `export default function MarketPage() {}`

**Barrel Files:**
- Create `index.ts` in component/hook/utility directories for clean imports
- Example in `src/components/index.ts`:
  ```typescript
  export { BetCard } from './BetCard';
  export { MarketHeader } from './MarketHeader';
  ```

## API Response Format

**Frontend consumption:**
- All API responses validated with Zod schemas
- Use strict typing for API contracts
- Example schema:
  ```typescript
  const BetResponseSchema = z.object({
    id: z.string().uuid(),
    marketId: z.string().uuid(),
    amount: z.number().positive(),
    status: z.enum(['pending', 'active', 'resolved'])
  });
  ```

**Backend responses:**
- Use consistent JSON structure: `{ data: {...}, status: "success" }`
- Include metadata: `{ data: {...}, pagination: { page: 1, total: 100 } }`
- Error responses: `{ detail: "error message", status_code: 400 }`

## Async Patterns

**Frontend:**
- Use async/await, not .then() chains
- Wrap async operations with loading/error state
- Example:
  ```typescript
  const [isLoading, setIsLoading] = useState(false);
  const placeBet = async (data: BetData) => {
    setIsLoading(true);
    try {
      await api.placeBet(data);
    } finally {
      setIsLoading(false);
    }
  };
  ```

**Backend:**
- Use async route handlers with `async def`
- Use SQLAlchemy async session: `AsyncSession`
- Always await database operations
- Example:
  ```python
  @app.post("/bets")
  async def create_bet(bet_data: BetCreate, session: AsyncSession):
      result = await session.execute(select(User).where(User.id == bet_data.user_id))
      user = result.scalar_one_or_none()
  ```
