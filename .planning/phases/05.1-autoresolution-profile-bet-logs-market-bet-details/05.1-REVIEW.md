# Phase 05.1: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 17
**Status:** issues_found

## Summary

This phase adds auto-resolution wiring (Open-Meteo weather source), a public transaction ledger, market participant/payout endpoints, and market detail UI panels. The overall quality is good — the backend schemas validate properly, the UNION ledger query is sound, and the i18n strings are consistent across all three locales.

Five warnings were found, primarily around correctness edge cases: an N+1 query loop in `list_markets`, sorting on string-typed numeric columns in the ledger, an insecure HTTP link in `profile/[username]/page.tsx`, a missing "load more" reset when sort changes, and an empty city value that the backend will silently reject. Six info-level items cover dead keys, missing null guards, and an unchecked `autoCity` guard.

No critical security vulnerabilities were found.

## Warnings

### WR-01: N+1 query loop in `list_markets` — 4–5 queries per market row

**File:** `backend/app/services/market_service.py:196-220`
**Issue:** For every market row returned, `list_markets` executes 4 separate `await db.execute(...)` calls (`get_bet_odds`, `position_count`, `comment_count`, `_get_choice_counts`) plus one for `upvote_count`. With the default page size of 20, this is 80–100 round-trips per request. This is not a performance-only concern: under load the query count can cause timeouts and partial failures, making the endpoint unreliable.
**Fix:** Batch all per-market aggregates in a single SQL query using subqueries or CTEs alongside the main `Bet` query, or at minimum use `asyncio.gather` to parallelize the per-row awaits:
```python
# Example: parallel fetch for a single row
odds, pos_count, cmt_count, choice_c, upv_count = await asyncio.gather(
    get_bet_odds(db, row.id),
    db.execute(select(func.count(BetPosition.id)).where(...)),
    db.execute(select(func.count(Comment.id)).where(...)),
    _get_choice_counts(db, row.id),
    db.execute(select(func.count(BetUpvote.bet_id)).where(...)),
)
```

### WR-02: Ledger `sort_by=bp` and `sort_by=tp` sort on string columns, not numeric

**File:** `backend/app/services/ledger_service.py:78-84`
**Issue:** `bp_delta` and `tp_delta` are cast to `String` in the UNION sub-queries (lines 47, 57–58, 62–68). When `sort_by="bp"` or `sort_by="tp"` is used, the ORDER BY applies to string representations (e.g. "10" < "9" lexicographically). This produces incorrect sort order for any non-trivial amounts and is a correctness bug exposed by the `test_transactions_sort_params` test which only checks status 200, not order.
**Fix:** Cast the delta columns to a numeric type in the UNION, or apply a numeric cast in the sort expression:
```python
from sqlalchemy import cast
from sqlalchemy.types import Numeric

# In sub-queries, cast to Numeric instead of String for the delta columns:
cast(BpTransaction.amount, Numeric).label("bp_delta"),
# Then the ORDER BY will sort numerically.
```

### WR-03: `autoCity` empty string is not validated before submit in `new/page.tsx`

**File:** `frontend/src/app/(protected)/markets/new/page.tsx:62-69`
**Issue:** When `autoResolution` is true, the `resolution_source` payload is built with `location: autoCity`. If the user clicks submit before typing in the city field (`autoCity === ""`), the frontend sends `"location": ""`. The backend `validate_resolution_source` validator checks `if not src.get("location")` (line 67 of `schemas/market.py`), which correctly returns 422, but the UI shows a generic `create.error_create` message instead of a field-specific error, and there is no `required` HTML attribute on the city `<input>` that would prevent form submission.
**Fix:** Add client-side guard in `onSubmit` before building the payload:
```typescript
if (autoResolution && marketType === "binary" && !autoCity.trim()) {
  setError(t("create.city_hint")); // or a dedicated error key
  setSubmitting(false);
  return;
}
```
Or add `required` to the city `<input>` (it currently has `required` but it is inside a conditionally rendered block — verify the form's `required` propagation).

### WR-04: Profile page uses bare `<a href>` for market links (not Next.js `<Link>`)

**File:** `frontend/src/app/(protected)/profile/[username]/page.tsx:324`
**Issue:** The transaction ledger renders market links as `<a href={/markets/${tx.market_id}}>`. In Next.js App Router this triggers a full page reload instead of client-side navigation, breaking the SPA experience and potentially losing query client cache.
**Fix:**
```tsx
import Link from "next/link";
// ...
<Link href={`/markets/${tx.market_id}`} className="hover:underline text-blue-600 dark:text-blue-400">
  {tx.market_title}
</Link>
```

### WR-05: `allTx` accumulation is not reset when sort direction/field changes simultaneously

**File:** `frontend/src/app/(protected)/profile/[username]/page.tsx:84-93`
**Issue:** `handleSort` calls `setAllTx([])` and `setTxOffset(0)` together. Both are state updates, but because `setTxOffset(0)` triggers a re-fetch and the `useEffect` condition is `txOffset === 0` — on the *first* sort change this works. However, if the user sorts the same field again (toggling direction), `txOffset` is already 0 and the effect dependency `[transactionsQuery.data, txOffset]` does not fire a fresh replacement because `txOffset` did not change. `setAllTx([])` clears it momentarily but the stale cache immediately repopulates via the effect before the new fetch completes.
**Fix:** Include `sortBy` and `sortDir` in the reset condition:
```typescript
useEffect(() => {
  if (transactionsQuery.data?.transactions) {
    setAllTx(transactionsQuery.data.transactions); // always replace on new query data
  }
}, [transactionsQuery.data]); // Remove txOffset dependency; rely on queryKey change
```
The query key already includes `[..., txOffset, sortBy, sortDir]` so a sort direction toggle triggers a new fetch; the effect just needs to replace (not append) whenever fresh data arrives and `txOffset === 0`.

## Info

### IN-01: `resolution_source` is stored as JSON string but never returned in `MarketResponse`

**File:** `backend/app/services/market_service.py:35`
**Issue:** `resolution_source` is serialized to JSON string on write (`json.dumps(data.resolution_source) if data.resolution_source else None`) but `MarketResponse` schema has no `resolution_source` field. This is fine for now, but any future UI that needs to show or edit the auto-resolution config will have no way to retrieve it without a schema change.
**Fix:** Add an optional field to `MarketResponse` if future display is needed:
```python
resolution_source: dict | None = None
```

### IN-02: `_get_current_user` is duplicated between `routes/markets.py` and `routes/users.py`

**File:** `backend/app/api/routes/markets.py:27-31`, `backend/app/api/routes/users.py:18-22`
**Issue:** Identical helper function defined in two route modules. A dedicated `deps.py` already exists — move this helper there.
**Fix:** Add to `app/api/deps.py` and import from both routers.

### IN-03: Participants table row key is fragile for multiple bets at same timestamp

**File:** `frontend/src/app/(protected)/markets/[id]/page.tsx:553`
**Issue:** `key={${p.user_id}-${p.created_at}}` — if the same user somehow has two positions at the exact same millisecond (edge case), React will warn about duplicate keys. Using position index or a position ID would be more robust.
**Fix:** Use a composite that includes `bp_staked` or request that the backend expose a position `id` field:
```tsx
key={`${p.user_id}-${p.created_at}-${p.bp_staked}`}
```

### IN-04: `test_autoresolution.py` registers the same `restest` username in every test — tests will fail if run in non-isolated DB state

**File:** `backend/tests/test_autoresolution.py:23-35`
**Issue:** All six tests share `_register_and_login` which registers `restest@example.com`. If tests run without per-test DB teardown (depending on fixture scope), the second test will get a 400/409 on register but the login still succeeds — masking the problem. Tests `test_resolution_source_accepted` and `test_resolution_source_persisted` both create markets with the same `BASE_MARKET` title, which may conflict on unique constraints.
**Fix:** Use unique suffixes or rely on the test fixture to provide fresh DB state. Confirm the `client` fixture provides per-test isolation.

### IN-05: `payoutsQuery` is gated on `status === "closed"` but the payout banner fires on `bet:resolved` for any status

**File:** `frontend/src/app/(protected)/markets/[id]/page.tsx:130-135`
**Issue:** `payoutsQuery` has `enabled: !!marketId && marketQuery.data?.status === "closed"`. The RT-03 `onBetResolved` handler (line 162) shows the banner and invalidates market data. If the market transitions to `"closed"` via the socket event, the payouts query will now be enabled — but the `invalidateQueries` on the market will trigger a refetch and enable the query correctly. This is fine in practice but depends on React Query processing state updates in the right order. If `marketQuery.data` hasn't been updated yet when `payoutsQuery`'s `enabled` is evaluated, the payout data won't load until the next render cycle.
**Fix:** This is low risk but consider adding a local state flag `const [marketJustResolved, setMarketJustResolved] = useState(false)` that is set to `true` in `onBetResolved` and included in the `enabled` expression.

### IN-06: Missing i18n keys in DE/FR for `"profile.settings"` tooltip and `"market.only_participants"`
