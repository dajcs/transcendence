# Phase 5.1: Autoresolution, Profile Bet Logs, Market Bet Details - Research

**Researched:** 2026-04-16
**Domain:** FastAPI backend endpoints + Next.js React Query frontend additions
**Confidence:** HIGH

## Summary

This phase is purely additive — no new tables, no migrations, no schema changes. All three work streams wire up already-existing backend logic or query already-existing tables via new endpoints.

**Stream 1 (Autoresolution):** The backend task `_fetch_open_meteo_outcome` already reads `resolution_source` JSON. The only missing piece is: (a) the `MarketCreate` schema does not accept `resolution_source`, and (b) the frontend form has no toggle. The JSON key the task reads is `location` (not `city`) and `provider` must be `"open-meteo"`. The frontend must serialize using those exact keys.

**Stream 2 (Profile Ledger):** `BpTransaction`, `TpTransaction`, `KpEvent` tables are fully populated. A UNION query in a new service function maps `reason` values to the UI `type` enum. Critical: `TpTransaction` has no `reason` field — its `type` is always inferred from context (`"bet_won"` if amount > 0). `KpEvent.source_type` provides the type string for KP events.

**Stream 3 (Market Bet Details):** Two new GET endpoints on `/api/markets/{id}`. Participant list queries `BetPosition` joined to `User`. Payout breakdown queries `BpTransaction` with `reason="bet_win"` and `TpTransaction` joined to `User`. Both are offset-paginated.

**Primary recommendation:** Implement as three independent plans in sequence — backend endpoints first (Plans 1 + 2), frontend additions second (Plan 3). No cross-stream blocking after endpoints exist.

### Pattern 3: UNION query for transaction ledger

```python
# Source: pattern derived from existing SQLAlchemy 2 usage in codebase [VERIFIED]
from sqlalchemy import union_all, literal, cast, String, select, func
from sqlalchemy.types import Uuid  # SQLite-compatible — NOT PG_UUID from sqlalchemy.dialects.postgresql

# BpTransaction sub-query
bp_q = select(
    BpTransaction.id,
    BpTransaction.created_at.label("date"),
    cast(BpTransaction.reason, String).label("reason"),
    BpTransaction.bet_id,
    BpTransaction.amount.label("bp_delta"),
    literal(0.0).label("tp_delta"),
).where(BpTransaction.user_id == user_id)

# TpTransaction sub-query
tp_q = select(
    TpTransaction.id,
    TpTransaction.created_at.label("date"),
    literal("bet_won").label("reason"),
    TpTransaction.bet_id,
    literal(0.0).label("bp_delta"),
    TpTransaction.amount.label("tp_delta"),
).where(TpTransaction.user_id == user_id)

# KpEvent sub-query (bet_id = null — KpEvent has source_id, not bet_id)
# Use cast(None, Uuid) from sqlalchemy.types — NOT cast(None, PG_UUID) from dialects.postgresql
kp_q = select(
    KpEvent.id,
    KpEvent.created_at.label("date"),
    KpEvent.source_type.label("reason"),
    cast(None, Uuid).label("bet_id"),
    literal(0.0).label("bp_delta"),
    literal(0.0).label("tp_delta"),
).where(KpEvent.user_id == user_id)

combined = union_all(bp_q, tp_q, kp_q).subquery()
```

**Critical:** KpEvent does not have a `bet_id` column — source_id is a generic UUID for any source, not a bet FK. The `market_title` for KP rows must be `None`.

### Pattern 5: React Query with "load more" accumulation

The UI-SPEC requires appending rows (not replacing). Use `useState` accumulation pattern (simpler than `useInfiniteQuery` for this codebase):

```typescript
const [offset, setOffset] = useState(0);
const [allRows, setAllRows] = useState<ParticipantRow[]>([]);

const query = useQuery({
  queryKey: ["market-positions", marketId, offset],
  queryFn: async () => (await api.get(`/api/markets/${marketId}/positions?offset=${offset}&limit=20`)).data,
  staleTime: 30_000,
});

useEffect(() => {
  if (query.data?.participants) {
    setAllRows(prev => offset === 0 ? query.data.participants : [...prev, ...query.data.participants]);
  }
}, [query.data, offset]);
```

### Anti-Patterns to Avoid

- **Using `"city"` key in resolution_source JSON:** The backend reads `"location"` — wrong key silently returns `None` outcome (Tier 1 falls through to Tier 2).
- **Using `"open_meteo"` (underscore) as provider:** The backend checks `provider == "open-meteo"` (hyphen).
- **Client-side sort on ledger:** UI-SPEC says re-fetch with `sort_by` + `sort_dir` query params — not in-memory sort.
- **Querying `KpEvent.bet_id`:** The column doesn't exist. KP events have `source_id` which is a generic FK.
- **Blocking positions endpoint on auth:** D-07/D-08 say participant list and ledger are fully public — no auth required on these GET endpoints.
- **Including `resolution_source` in Bet.resolution_source as a Python dict:** The column type is `Text` (not JSONB) — serialize to `json.dumps()` before storing.
- **Using `PG_UUID` from `sqlalchemy.dialects.postgresql` for NULL cast:** Breaks SQLite test environment. Use `Uuid` from `sqlalchemy.types` instead.

## Common Pitfalls

### Pitfall 1: `resolution_source` JSON key mismatch
**What goes wrong:** City typed by user never geocodes — Tier 1 silently fails, market escalates to proposer.
**Why it happens:** `_fetch_open_meteo_outcome` reads `source.get("location")` but UI-SPEC component comment says "city". The key name in the stored JSON must be `location`.
**How to avoid:** `MarketCreate.resolution_source` schema should validate/normalize the key, or the frontend must use `{ location: cityValue }` not `{ city: cityValue }`.
**Warning signs:** Auto-resolved markets always go to `pending_resolution` instead of `proposer_resolved`.

### Pitfall 2: `resolution_source` is `Text`, not `JSONB`
**What goes wrong:** Storing a dict directly raises SQLAlchemy type error.
**Why it happens:** `Bet.resolution_source: Mapped[str | None]` is a Text column (line 20 of bet.py).
**How to avoid:** `market_service.create_market` must call `json.dumps(data.resolution_source)` before assigning to `bet.resolution_source`. The `MarketCreate` schema accepts `dict | None`; the service serializes it.
**Warning signs:** `AttributeError` or Postgres type error at market creation.

### Pitfall 3: KpEvent has no bet_id
**What goes wrong:** UNION query crashes because KpEvent has no `bet_id` column.
**Why it happens:** `KpEvent` has `source_id` (generic UUID), not a FK to bets.
**How to avoid:** UNION sub-query for KP rows must use `cast(None, Uuid).label("bet_id")` where `Uuid` is from `sqlalchemy.types` (SQLite-compatible). Do NOT use `PG_UUID` from `sqlalchemy.dialects.postgresql`.
**Warning signs:** `OperationalError: column "bet_id" does not exist` on KpEvent table.

### Pitfall 4: Payout data source
**What goes wrong:** Payout breakdown shows nothing for closed markets.
**Why it happens:** Payout data is in `BpTransaction` (reason="bet_win") and `TpTransaction` — not a separate payout table. The query must JOIN these tables to User on `user_id`, filtered by `bet_id`.
**How to avoid:** `GET /api/markets/{id}/payouts` queries `BpTransaction WHERE reason="bet_win" AND bet_id=id` JOINED to User, plus `TpTransaction WHERE bet_id=id` JOINED to User. Merge by user_id in Python.
**Warning signs:** Empty payout table on closed markets you know were resolved.

### Pitfall 5: font-bold in new JSX
**What goes wrong:** UI-SPEC checker rejects the implementation.
**Why it happens:** UI-SPEC explicitly bans `font-bold` (700) — only `font-semibold` (600) and `font-medium` (500) allowed.
**How to avoid:** Search all new JSX for `font-bold` before committing. The existing `page.tsx` files use `font-bold` in older sections — don't copy that pattern.

### Pitfall 6: Auto-resolution toggle with non-binary market type
**What goes wrong:** User selects multiple_choice AFTER enabling auto-resolution — sub-panel remains visible.
**Why it happens:** React state is not reset on market type change.
**How to avoid:** In the `onClick` handler for non-binary market type buttons, also call `setAutoResolution(false)`.

### Pitfall 7: Python-slice pagination in get_market_payouts
**What goes wrong:** `bp_rows = (await db.execute(...)).all()` loads ALL payout rows into Python memory before slicing — O(N) memory for large markets.
**Why it happens:** Using `bp_rows[offset: offset + limit]` after fetching all rows.
**How to avoid:** Apply `.offset(offset).limit(limit)` at the SQLAlchemy query level. Use a separate `COUNT(DISTINCT user_id)` subquery for `total` instead of `len(bp_rows)`.

### Adding resolution_source to MarketCreate schema

```python
# backend/app/schemas/market.py — add to MarketCreate
from typing import Any
import json

class MarketCreate(BaseModel):
    # ... existing fields ...
    resolution_source: dict[str, Any] | None = None

    @model_validator(mode="after")
    def validate_resolution_source(self) -> "MarketCreate":
        if self.resolution_source is not None:
            if self.market_type != "binary":
                raise ValueError("Auto-resolution only supported for binary markets")
            src = self.resolution_source
            if src.get("provider") != "open-meteo":
                raise ValueError("Only 'open-meteo' provider supported")
            if not src.get("location"):
                raise ValueError("resolution_source.location is required")
            if src.get("condition") not in ("rain", "temperature"):
                raise ValueError("condition must be 'rain' or 'temperature'")
        return self
```

### Payout breakdown query pattern (SQL-level pagination)

```python
# backend/app/api/routes/markets.py — get_market_payouts
# Step 0: Count total distinct BP winners
total = (await db.execute(
    select(func.count(func.distinct(BpTransaction.user_id)))
    .where(BpTransaction.bet_id == market_id, BpTransaction.reason == "bet_win")
)).scalar_one()

# Step 1: BP winners with SQL-level pagination (NOT Python slice)
bp_rows = (await db.execute(
    select(BpTransaction.user_id, User.username, func.sum(BpTransaction.amount).label("bp_won"))
    .join(User, User.id == BpTransaction.user_id)
    .where(BpTransaction.bet_id == market_id, BpTransaction.reason == "bet_win")
    .group_by(BpTransaction.user_id, User.username)
    .offset(offset).limit(limit)
)).all()

# Step 2: TP winners (separate query, merge by user_id)
tp_rows = (await db.execute(
    select(TpTransaction.user_id, func.sum(TpTransaction.amount).label("tp_won"))
    .where(TpTransaction.bet_id == market_id)
    .group_by(TpTransaction.user_id)
)).all()
tp_map = {row.user_id: float(row.tp_won) for row in tp_rows}
```

## Open Questions (RESOLVED)

1. **Which `KpEvent.source_type` values to include in the ledger?**
   - What we know: `source_type` values found in codebase: `"comment_upvote"`, `"kp_reset_login"`, `"daily_allocation"`
   - What's unclear: Should `kp_reset_login` (negative KP reset) appear as a ledger entry? It's noise.
   - RESOLVED: Include only `KpEvent` rows where `amount > 0` — excludes resets. Implemented via `.where(KpEvent.amount > 0)` in the UNION sub-query (Plans 02, ledger_service.py). The BP gain from KP conversion appears as a `BpTransaction(reason="kp_conversion")` and is more user-meaningful.

2. **`TpTransaction.bet_id` is nullable in the schema but marked non-nullable in the model**
   - What we know: Model line 28 `bet_id: Mapped[uuid.UUID]` with `nullable=False`. All TP transactions are bet-related. Safe to assume `bet_id` is always present.
   - What's unclear: Is there any TP created without a bet_id (e.g., signup bonus)?
   - RESOLVED: Treat `bet_id` as always present for TP rows; include market title lookup for all TP entries. Model constraint (`nullable=False`) confirms this. Implemented in ledger_service.py.
