# Phase 6.1: Like Points Rename + Economy Formula Fix - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Rename Karma Points (KP) to Like Points (LP) everywhere — DB, backend, frontend, i18n, docs.
Audit and fix all BP/TP earn/win formulas: LP→BP conversion cap, bet cap (now flat), TP time-based payout, binary/multichoice 10x max payout, numeric market payout overhaul, market creator rebate removal, and special fund tracking for surplus.

</domain>

<decisions>
## Implementation Decisions

### KP → LP Rename (Full Rename)
- **D-01:** Full rename at all layers:
  - DB table: `kp_events` → `lp_events` (Alembic migration 015)
  - Python class: `KpEvent` → `LpEvent` (and all imports)
  - Backend API response field: `kp` → `lp` (wherever `get_balance()` or user schemas return it)
  - Frontend Zustand store: `kp: number` → `lp: number` in `auth.ts`
  - All i18n strings updated: "Karma Points" → "Like Points", "KP" → "LP", "KP→BP" → "LP→BP", etc.
- **D-02:** All docs updated: `plan/ECONOMY.md`, `CLAUDE.md`, `README`, `REQUIREMENTS.md`, and any other file mentioning "Karma Points" or "KP".

### Heart UI (Frontend)
- **D-03:** Nav bar displays ♥ N (red heart emoji + count) instead of "LP N" text. Format: `♥ {lp_count}` between BP and TP values.
- **D-04:** Comment and market upvote buttons become heart icons — empty heart when not liked, filled red heart when liked (Twitter-style toggle).
- **D-05:** Like state is persisted per user. Heart renders filled/empty based on whether the current user has already upvoted (existing upvote tracking). Clicking a filled heart removes the upvote (unlike).
- **D-06:** Red heart, no animation (instant toggle).
- **D-07:** LP balance behavior unchanged from KP: accumulates from upvotes, resets to 0 on login after LP→BP conversion.

### LP → BP Conversion Formula Fix
- **D-08:** Formula: `min(log2(lp + 1), 10.0)` — full float, **capped at 10 BP**.
  Current code (`math.log2(kp_value + 1)`) is missing the cap. Fix `convert_kp_to_bp()` in `economy_service.py`.
  - 0 LP → 0.0 BP
  - 1 LP → 1.0 BP
  - 2 LP → 1.585 BP
  - 1023 LP → 10.0 BP
  - 2000 LP → 10.0 BP (capped)

### Bet Cap — Flat 10 BP
- **D-09:** Remove the `compute_bet_cap()` log10 formula. New rule: flat cap of **10 BP per position** (or available balance, whichever is lower). No formula.
- **D-10:** Bet placement UI updated — default bet amount is 1 BP. Dropdown arrow opens a list with discrete options: 1, 2, 3, … min(10, floor(available_balance)). Backend enforces the 10 BP cap and available-balance check.

### TP Formula — Time-Based (Fix Code)
- **D-11:** Fix `_compute_tp_for_user()` in `resolution_service.py` to use the time-based formula: `tp = t_win / t_bet` (raw float, **no truncation**). `t_win` = seconds the user held the winning position; `t_bet` = total bet duration.
  - The existing `compute_tp_earned(t_win, t_bet)` function is the correct formula but is currently unused in payout. Wire it in (remove the floor truncation, use raw float).
  - The old stake-proportional formula (`floor(bp_staked / total_winning_stake * 100) / 100`) is replaced.

### Binary / Multichoice Payout — 10x Max Multiplier
- **D-12:** Max payout per winner = `bet_amount × 10` (hard cap). Old rule (distribute full pool proportionally) is replaced.
  - Each winner receives `min(bet_amount × 10, their_proportional_share)`.
  - Surplus BP (pool remainder after paying all winners at cap) goes to the **special fund**.
- **D-13:** Remove market creator rebate. The current `+1.0 bp` rebate for voting on own market is revoked. Market creators pay 1 BP to bet like any other participant.

### Numeric Market Payout — New Formula
- **D-14:** If a prediction matches the resolution number exactly → winner gets `bet_amount × 10` (10x multiplier).
- **D-15:** If nobody predicted exactly → expand the winning gap symmetrically around the resolution number until `ceil(total_bettors × 0.10)` participants fall inside the window.
- **D-16:** Within the winning window: linear multiplier interpolation — closest prediction gets 10x, farthest (but still in window) gets 0.1x. Formula: `multiplier = 10 - (rank / (window_size - 1)) × 9.9` where rank 0 = closest.
- **D-17:** Surplus BP (pool remainder after paying all windowed winners) goes to the **special fund**.

### Special Fund — New DB Table
- **D-18:** New table (e.g., `bp_fund_entries`) tracks surplus per payout event:
  - Columns: `id`, `market_id`, `user_id` (the winner whose payout was capped), `amount` (surplus that would have been theirs), `reason` (e.g., `"cap_surplus"`, `"numeric_surplus"`), `created_at`.
  - Recorded atomically within the same payout transaction.

### Claude's Discretion
- Exact Alembic migration column renaming approach (alter vs create-new, if any).
- Heart icon component library choice (emoji vs SVG icon from existing icon set).
- Whether `compute_bet_cap()` function is deleted or kept as dead code with a deprecation comment.
- The exact SQL for the numeric winning-window expansion algorithm.

</decisions>
