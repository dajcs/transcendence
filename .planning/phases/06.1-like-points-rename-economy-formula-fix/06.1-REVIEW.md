# Phase 06.1: Code Review Report

**Reviewed:** 2026-04-20
**Depth:** standard
**Files Reviewed:** 21
**Status:** issues_found

## Summary

This phase renames KP→LP across the full stack, adds LP toggle (unlike) endpoints, rewrites payout formulas with a 10x cap, and introduces the `bp_fund_entries` surplus ledger. The migration files and schema changes are sound. The core rename is complete and consistent.

Four warnings were found: a missing self-upvote guard on markets (asymmetric with the comment service), a race condition in both unlike paths that can double-decrement LP, zero TP awarded to all numeric market winners due to a side-string mismatch, and a bet amount selector that presents an unaffordable option when BP < 1. Five informational items cover minor inconsistencies.

## Warnings

### WR-01: Missing Self-Upvote Guard on `upvote_market` and `unlike_market`

**File:** `backend/app/services/market_service.py:327`
**Issue:** `upvote_market` lets the proposer heart their own market, awarding themselves 1 LP. The comment service explicitly guards against this (`if comment.user_id == voter_id: return`). The omission is asymmetric. `unlike_market` also lacks the symmetric guard, though the practical impact there is smaller. Bounded only by the unique constraint, which prevents double-award — but the first click goes through.
**Fix:**
```python
async def upvote_market(db: AsyncSession, user_id: uuid.UUID, market_id: uuid.UUID) -> None:
    market = (await db.execute(select(Bet).where(Bet.id == market_id))).scalar_one_or_none()
    if market is None:
        raise HTTPException(status_code=404, detail="Market not found")
    if market.proposer_id == user_id:
        return  # self-upvote — silent no-op (mirrors comment service)
    # ... rest of function unchanged
```
Apply the same guard to `unlike_market`.

### WR-02: Race Condition in Unlike Paths — Double LP Decrement

**File:** `backend/app/services/comment_service.py:153`, `backend/app/services/market_service.py:349`
**Issue:** Both `unlike_comment` and `unlike_market` follow the pattern:

```
SELECT upvote row
→ if None: return
→ DELETE row
→ INSERT LpEvent(amount=-1)
→ commit
```

There is no `SELECT ... FOR UPDATE` lock on the upvote row and no uniqueness constraint that would cause the second concurrent transaction to fail (unlike the upvote path, which is protected by an `IntegrityError` on the unique constraint). Two rapid identical DELETE requests (e.g., double-tap, two tabs) can both pass the `if upvote is None` check before either has committed, both delete (second is a no-op), and both insert a `-1` LpEvent. The net result is `-2 LP` for one logical unlike.

**Fix:** Add a `with_for_update()` lock on the upvote select, or rely on the database row-level lock:
```python
upvote = (
    await db.execute(
        select(CommentUpvote)
        .where(
            CommentUpvote.comment_id == comment_id,
            CommentUpvote.user_id == voter_id,
        )
        .with_for_update()  # row-level lock prevents double-unlike
    )
).scalar_one_or_none()
if upvote is None:
    return
await db.delete(upvote)
# ... LpEvent insert and commit
```
Apply identically to `unlike_market`.

### WR-03: Numeric Market Winners Receive Zero TP

**File:** `backend/app/services/resolution_service.py:79`
**Issue:** `_compute_t_win` filters `PositionHistory.side == winning_side`. For numeric markets, `winning_side` is the resolution value string (e.g. `"42.5"`). But each bettor's position `side` is their own prediction string (`"41.7"`, `"43.9"`, etc.). An exact string equality match will almost never hit — the only case where it returns a non-zero value is if a bettor predicted the exact resolved value and it is also recorded in `PositionHistory` with that exact string.

The downstream effect is that `_compute_tp_for_user` returns `0.0` for virtually all numeric market winners (line 256-258), so no `TpTransaction` is written and numeric bettors never earn TP. If zero TP for numeric bettors is an intentional design choice it should be documented; otherwise the fix is to skip the `side` filter for numeric markets:

```python
async def _compute_t_win(
    db, bet_id, user_id, winning_side, deadline, market_type="binary"
) -> float:
    q = select(PositionHistory.changed_at).where(
        PositionHistory.bet_id == bet_id,
        PositionHistory.user_id == user_id,
    )
    if market_type != "numeric":
        q = q.where(PositionHistory.side == winning_side)
    # take most recent entry as bet entry time
    q = q.order_by(PositionHistory.changed_at.asc()).limit(1)
    entry_time = (await db.execute(q)).scalar_one_or_none()
    # ... rest unchanged
```

### WR-04: Bet Amount Dropdown Offers Unaffordable Option When BP < 1

**File:** `frontend/src/app/(protected)/markets/[id]/page.tsx:1159`
**Issue:** The dropdown is generated as:
```tsx
Array.from(
  { length: Math.max(1, Math.min(10, Math.floor(currentUser?.bp ?? 1))) },
  (_, i) => i + 1
)
```
When `currentUser.bp` is between 0 and 0.99, `Math.floor` produces 0, but `Math.max(1, 0)` bumps the length to 1, rendering a "1 BP" option. The user has less than 1 BP, so placing that bet will be rejected by the backend. The `?? 1` fallback when `currentUser` is null has the same issue.

**Fix:**
```tsx
const maxBet = currentUser ? Math.min(10, Math.floor(currentUser.bp)) : 0;
// Only render the form if maxBet > 0
{maxBet > 0 && (
  <div className="mb-3 flex items-center gap-2">
    <label ...>{t("market.bet_amount_label")}</label>
    <select ...>
      {Array.from({ length: maxBet }, (_, i) => i + 1).map((n) => (
        <option key={n} value={n}>{n} BP</option>
      ))}
    </select>
  </div>
)}
```

### IN-02: Inconsistent Numeric Formatting of BP/TP in Dashboard vs TopNav

**File:** `frontend/src/app/(protected)/dashboard/page.tsx:88`
**Issue:** Dashboard renders `{user.bp}` and `{user.tp}` as raw numbers. TopNav uses `{(user?.bp ?? 0).toFixed(1)}` and `{(user?.tp ?? 0).toFixed(1)}`. Users may see `"1.0000000001"` or similar floating-point artifacts in the dashboard panel.
**Fix:** Apply `.toFixed(1)` or `.toFixed(2)` consistently: `BP: {user.bp.toFixed(1)} · LP: {user.lp} · TP: {user.tp.toFixed(1)}`

### IN-03: Silent Loss of BP for Unparseable Positions in Numeric Payout

**File:** `backend/app/services/resolution_service.py:125`
**Issue:** `_compute_numeric_payouts` computes `total_bp_pool` from all positions but then silently drops any position where `float(side_str)` fails (line 127-129). That user's BP enters the pool but they receive zero payout and zero surplus ledger entry — the BP vanishes into the aggregate surplus without attribution. In practice numeric sides should always be parseable, but the silent-drop pattern means a malformed `side` string results in unrecoverable BP loss with no log.
**Fix:** Log a warning or add a `BpFundEntry` for unparseable entries so they are accounted for. At minimum, raise an explicit error if `parsed` is empty while `positions` is not.

### IN-04: Stale `kp_` Naming in i18n Keys and Frontend Color Map

**File:** `frontend/src/app/(protected)/profile/[username]/page.tsx:60`, `frontend/src/i18n/en.ts:209,284`
**Issue:** After the KP→LP rename, three identifiers still carry the old name:
- `TYPE_COLORS["kp_allocation"]` in the profile page (maps a transaction type that may be emitted by the backend)
- i18n key `"ledger.type_kp_allocation"` (en/fr/de)
- i18n key `"notif.kp_converted"` (en/fr/de)

The display values correctly say "LP→BP" and "LP Converti", so the rename is cosmetically complete. However, if the backend emits `reason="lp_allocation"` after a future rename, the `TYPE_COLORS` map will fail to find a matching color and fall back to the default. 
**Fix:** Either rename these keys to `lp_allocation` / `notif.lp_converted` (breaking change), or add an alias key so both old and new names resolve:
```ts
"ledger.type_lp_allocation": "LP→BP",
```

### IN-05: `Market` Type Has Orphan `"pending"` Status Value

**File:** `frontend/src/lib/types.ts:7`
**Issue:** The `Market` interface includes `"pending"` in the status union: `"open" | "pending" | "pending_resolution" | ...`. The backend `Bet.status` does not emit `"pending"` — it uses `"pending_resolution"`. This dead union member causes no runtime error but can mislead callers who check `if (market.status === "pending")`.
**Fix:** Remove `"pending"` from the union, or confirm it is used by some other backend path not covered in this review.
